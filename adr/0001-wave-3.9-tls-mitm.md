# ADR-0001 · Wave 3.9 TLS-MITM Egress Inspection

- **Date:** 2026-07-15
- **Status:** DECIDED — Option 1 (Adapter-First, Hostname-Only Proxy) approved. Option 2 (opt-in MITM) and Option 3 (default MITM) rejected.
- **Owners:** Runtime + Access Model WG
- **Related:** internal design notes (not public).

## Context

To generate rich AVAR receipts and enforce content-aware policy obligations, Aarmos benefits from visibility into agent payload contents (prompts, tool arguments, model responses). A prior proposal (Wave 3.9) suggested an opt-in TLS Man-In-The-Middle (MITM) proxy: install a locally-generated root CA, terminate TLS at the local daemon, inspect the plaintext, then re-encrypt on the way out.

Our first-class client adapters — `@aarmos/adapter-mcp`, `@aarmos/adapter-openapi`, `@aarmos/adapter-deeplink`, and the LLM adapters used in `examples/langchain`, `examples/crewai`, `examples/autogen` — already capture prompt/response telemetry pre-encryption for all traffic that flows through them. The share of real-world agent traffic that uses adapters vs. raw HTTPS is **not yet measured** (we have zero users). The MITM path additionally carries three concrete costs: SDK cert-pinning breakage, 2026 regulatory exposure (post-EU AI Act Art. 14; wiretap-adjacent disclosure risk), and a permanent tension with our locked "local-first / Switzerland" positioning.

## Decision

1. The L7 network proxy operates strictly via **hostname-only HTTPS `CONNECT` tunneling**. It filters and receipts outbound FQDNs; it does not decrypt payloads.
2. Rich payload provenance (prompt, args, response) is captured **inside the client-side adapters and the bridge layer**, before encryption, on the paths users opt into by importing our adapters.
3. Un-adapted "escape-hatch" HTTPS is allowed or denied at the FQDN level only. Its limited receipt fidelity is disclosed transparently.
4. The `aarmos trust` CA-install command remains in the CLI as dormant surface for future opt-in features (e.g. per-tenant DLP). It is **not** wired to any current enforcement path and is not advertised as a shipped capability.

## Consequences

**Positive**
- No root CA install required to get value from Aarmos. Preserves Switzerland/local-first invariants.
- No wiretap-adjacent legal surface. No cert-pinning breakage for SDKs (OpenAI, Anthropic, Stripe, etc.).
- Incentive alignment: users get high-fidelity receipts by using our adapters. Ergonomics, not coercion.
- Wave 3.9 is closed, not parked. Wave 3.10 (Contracts) and downstream waves unblock cleanly.

**Negative / accepted**
- Any HTTPS call from user agent code that bypasses our adapters produces a low-fidelity receipt: `{ fqdn, timestamp, bytes, receipt_id, tool_id? }` and nothing about the payload.
- We cannot claim "full payload provenance" as an unconditional property. Copy must scope claims to adapter-mediated traffic.
- Content-aware policy obligations (e.g. arg-level DLP) only fire on the adapter path.

## Revisit Triggers

This decision is reopened only if **either** condition is met:

1. **Empirical bypass signal.** Live dogfooding shows >20% of critical agent tool-execution traffic bypasses our first-class adapters via raw HTTPS, measured across ≥1,000 recorded sessions from ≥5 distinct dogfood teams (aligned with the P1 exit criteria in `EGRESS-ROADMAP.md`).
2. **Commercial signal.** A paying design partner makes payload-level DLP a contractual condition of the deal.

Absent either signal, MITM stays off the roadmap.

## Explicitly rejected alternatives

- **Option 2 — Opt-in MITM (off by default).** Rejected: still requires shipping and supporting cert-generation, per-SDK pinning workarounds, uninstall paths, and the legal disclosure surface. The support tax is roughly the same as Option 3 while serving <20% of the value.
- **Option 3 — MITM on by default.** Rejected: violates every locked invariant (Switzerland, local-first, no-overpromise).
